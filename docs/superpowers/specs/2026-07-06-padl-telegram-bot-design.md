# Спецификация: Telegram-бот для мониторинга свободных слотов ПАДЛ

Дата: 2026-07-06
Статус: готово к ревью

## 1. Цель

Нужно создать программу, которая работает только в Trigger.dev и каждые 60 секунд отправляет в Telegram-бот текущее состояние доступных слотов ПАДЛ с сайта `https://outdoor.sport.mos.ru/#venues-events`.

Каждое сообщение в Telegram отражает состояние слотов на момент проверки. Сообщение отправляется всем пользователям, которые запускали бота командой `/start`, даже если список слотов не изменился. Если подходящих слотов нет, отправляется сообщение `Свободных слотов сейчас нет`.

Важное требование по стоимости: программа не должна работать как бесконечный процесс. Каждый запуск Trigger.dev должен быстро выполнить одну проверку, отправить сообщения, сохранить состояние и завершиться. Следующий запуск создает уже cron-расписание Trigger.dev.

## 2. Ограничения

- Вся программа работает только в Trigger.dev.
- Не используются сторонние сервисы хранения данных: базы данных, Redis, Google Sheets, отдельные backend-серверы и webhook-хостинг не применяются.
- В Telegram-боте нет команд кроме `/start`.
- Все параметры поиска меняются через env-переменные.
- Плановый запуск выполняется раз в минуту.
- У scheduled-задачи обязательно должен быть `concurrency: 1`, чтобы два запуска не обновляли состояние подписчиков одновременно и не перетирали друг друга.
- Запрещены бесконечные циклы polling внутри процесса: `while (true)`, `setInterval`, рекурсивный self-trigger, `wait.for()` для минутного цикла и Telegram long polling.

## 3. Основной подход

Используется один основной scheduled task в Trigger.dev, например `padl-monitor`.

Каждый запуск задачи:

1. Загружает состояние бота из Trigger.dev Session metadata.
2. Получает новые Telegram-сообщения через `getUpdates`.
3. Добавляет в подписчики всех пользователей, отправивших `/start`.
4. Обновляет Telegram `offset` в локальном состоянии, чтобы следующий запуск не обрабатывал старые updates повторно.
5. Получает актуальные слоты ПАДЛ с сайта outdoor.sport.mos.ru.
6. Фильтрует слоты по параметрам из env.
7. Формирует компактное сообщение.
8. Отправляет сообщение всем подписчикам.
9. Сохраняет обновленное состояние обратно в Trigger.dev Session metadata.
10. Завершается.

Задача не ожидает следующую минуту внутри себя. Повторный запуск выполняется только Trigger.dev cron-расписанием `* * * * *`.

## 4. Хранение подписчиков

Подписчики хранятся внутри Trigger.dev в metadata одной служебной session.

Session:

- `externalId`: `padl-telegram-bot-state`
- `type`: например `padl.telegram-state`
- `metadata`: JSON с offset Telegram и списком подписчиков.

Если session еще не существует, первый запуск `padl-monitor` выполняет lazy-create служебной session через Trigger.dev Sessions API. Для создания используется отдельный no-op task, например `padl-state-session`, который нужен только как технический `taskIdentifier` для session и не содержит бизнес-логики мониторинга. После создания состояние читается и обновляется по `externalId`.

Формат состояния:

```json
{
  "telegramOffset": 123456789,
  "subscribers": {
    "111111111": {
      "chatId": 111111111,
      "firstSeenAt": "2026-07-06T12:00:00.000Z",
      "lastStartAt": "2026-07-06T12:00:00.000Z"
    }
  }
}
```

Ключом в `subscribers` является строковое значение `chatId`.

Если пользователь несколько раз отправляет `/start`, подписчик не дублируется. Обновляется только `lastStartAt`.

Если Telegram возвращает ошибку отправки, означающую что бот заблокирован или чат недоступен, подписчик удаляется из состояния при следующем сохранении. Отдельной команды отписки нет, потому что в боте разрешена только `/start`.

## 5. Telegram-интеграция

Бот работает через короткий polling-запрос, а не через webhook и не через бесконечный polling-loop.

Причина: polling позволяет работать без отдельного публичного HTTP endpoint и без внешнего сервера. Trigger.dev-задача сама раз в минуту вызывает Telegram Bot API.

Алгоритм:

1. Взять `telegramOffset` из состояния.
2. Вызвать `getUpdates` с параметрами:
   - `offset`: сохраненный offset;
   - `allowed_updates`: только `message`;
   - `timeout`: `0`.
3. Для каждого update проверить текст сообщения.
4. Если текст равен `/start` или начинается с `/start `, добавить `message.chat.id` в подписчики.
5. После обработки updates записать в новое состояние `telegramOffset = max(update_id) + 1`.

Telegram считает update подтвержденным только после следующего вызова `getUpdates` с offset выше его `update_id`. Поэтому offset нельзя считать надежно обработанным, пока новое состояние не сохранено в Trigger.dev Session metadata.

Сообщения, отличные от `/start`, игнорируются. Бот не отвечает на них отдельным текстом, чтобы не создавать скрытых команд или лишнего поведения.

`timeout: 0` обязателен: задача должна сразу получить уже накопленные updates и идти дальше. Нельзя использовать Telegram long polling с большим `timeout`, потому что он держит процесс Trigger.dev открытым в ожидании новых сообщений и расходует лимиты.

Пример жизненного цикла одного запуска:

```text
12:00:00 старт run
12:00:01 getUpdates timeout=0
12:00:03 получение слотов
12:00:05 отправка сообщений
12:00:06 сохранение state
12:00:06 завершение run
12:01:00 следующий run по cron
```

## 6. Получение данных о слотах

Источник данных - публичный сайт `outdoor.sport.mos.ru` и его backend API `https://api.outdoor.sport.mos.ru`.

Базовый алгоритм:

1. Получить список площадок:
   - `GET https://api.outdoor.sport.mos.ru/items/venues`
   - используются опубликованные площадки и поля `id`, `title`, `address`, `working_hours`, `tags`.
2. Получить карточки типов игр:
   - `GET https://api.outdoor.sport.mos.ru/items/event_cards`
   - используются `id`, `title`, `event_type`, `venue_id`, `events`, `ms_night`.
3. Для каждой подходящей пары `venue_id + event_type` получить доступные даты и корты через booking API:
   - `GET /booking/date-options`
   - `GET /booking/availability`
4. Из availability-ответов получить события и длительности, у которых:
   - слот включен;
   - бронирование открыто;
   - количество доступных мест строго равно env-параметру количества человек;
   - время слота попадает в env-диапазон;
   - тип игры и площадка соответствуют env-фильтрам.

Booking API может требовать `X-Booking-Access-Token`. Реализация должна сначала использовать HTTP-путь, повторяющий клиентский flow сайта для получения booking access token. Если этот flow потребует браузерной проверки, допускается headless browser внутри Trigger.dev как fallback. Такой fallback не нарушает ограничение, потому что не добавляет внешний сервис хранения или отдельный backend.

## 7. Env-переменные

Обязательные:

```bash
TELEGRAM_BOT_TOKEN=...
PADL_VENUES=all
PADL_TIME_FROM=10:00
PADL_TIME_TO=22:00
PADL_REQUIRED_PEOPLE=4
PADL_GAME_TYPES=free_play,masterclass,tournament_60,tournament_120,tournament_180
```

Операционные:

```bash
PADL_STATE_SESSION_EXTERNAL_ID=padl-telegram-bot-state
PADL_TIMEZONE=Europe/Moscow
PADL_CRON=* * * * *
PADL_MAX_MESSAGE_LENGTH=3900
TELEGRAM_GET_UPDATES_TIMEOUT_SECONDS=0
```

Правила env:

- `PADL_VENUES` принимает `all`, id площадок или точные названия через запятую.
- `PADL_TIME_FROM` и `PADL_TIME_TO` задаются в формате `HH:mm` по московскому времени.
- `PADL_REQUIRED_PEOPLE` задает строгое количество свободных мест. Подходит только слот, где доступных мест ровно столько же.
- `PADL_GAME_TYPES` задает типы игры через запятую. Ожидаемые значения берутся из `event_type`, например `free_play`, `masterclass`, `tournament_60`, `tournament_120`, `tournament_180`.
- Если env некорректный, задача должна завершаться с понятной ошибкой в логах и не отправлять пользователям частично неверные данные.
- `PADL_CRON` используется как значение cron-расписания при настройке Trigger.dev-задачи. Изменение cron в Trigger.dev может требовать redeploy или обновления schedule, поэтому runtime-фильтрами считаются только `PADL_VENUES`, `PADL_TIME_FROM`, `PADL_TIME_TO`, `PADL_REQUIRED_PEOPLE` и `PADL_GAME_TYPES`.
- `TELEGRAM_GET_UPDATES_TIMEOUT_SECONDS` должен быть `0` для короткого polling-запроса. Значения больше `0` допускаются только для ручной отладки, но не для production.

## 8. Фильтрация слотов

Слот попадает в сообщение только если выполнены все условия:

- площадка входит в `PADL_VENUES`;
- тип игры входит в `PADL_GAME_TYPES`;
- время начала слота находится в диапазоне `PADL_TIME_FROM <= slotTime <= PADL_TIME_TO`;
- слот доступен для бронирования;
- количество доступных мест строго равно `PADL_REQUIRED_PEOPLE`;
- дата и время интерпретируются в timezone `Europe/Moscow`.

Если API возвращает несколько длительностей для одного времени, каждая подходящая длительность считается отдельным вариантом, если в ней отличается количество мест, длительность или тип игры.

## 9. Формат Telegram-сообщения

Сообщение должно быть компактным.

Если слотов нет:

```text
Свободных слотов сейчас нет
```

Если слоты есть:

```text
Ст. метро Баррикадная
06.07 18:00 — 4 чел. — Свободная игра
06.07 19:00 — 4 чел. — Мастер-класс с инструктором

Ст. метро Третьяковская
06.07 20:00 — 4 чел. — Свободная игра
```

Правила форматирования:

- разные площадки разделяются пустой строкой;
- внутри площадки слоты сортируются по дате и времени;
- площадки сортируются по порядку из API или по порядку в `PADL_VENUES`, если задан конкретный список;
- если сообщение превышает `PADL_MAX_MESSAGE_LENGTH`, оно делится на несколько сообщений без разрыва строки одного слота.

## 10. Ошибки и устойчивость

Если не удалось получить слоты:

- ошибка логируется в Trigger.dev;
- состояние подписчиков не теряется;
- подписчикам отправляется короткое сообщение:

```text
Не удалось проверить слоты. Попробую снова через минуту.
```

Если не удалось отправить сообщение конкретному подписчику:

- ошибка логируется с `chatId`;
- при ошибках блокировки бота или недоступности чата подписчик удаляется;
- ошибки одного подписчика не должны останавливать рассылку остальным.

Если не удалось сохранить состояние:

- задача должна завершиться ошибкой, чтобы проблема была видна в Trigger.dev;
- Telegram `offset` нельзя считать надежно сохраненным до успешного обновления session metadata.

## 11. Concurrency и защита состояния

Задача должна быть настроена с `concurrency: 1`.

Это обязательное условие архитектуры. Без него два scheduled run могут одновременно:

- прочитать старое состояние;
- добавить разных подписчиков;
- сохранить разные версии metadata;
- потерять часть подписчиков или Telegram offset.

`concurrency: 1` гарантирует последовательную обработку run'ов одной задачи. Если один запуск еще выполняется, следующий не должен параллельно менять session metadata.

Если один запуск по какой-то причине длится дольше минуты, следующий запуск должен ждать в очереди или быть пропущен согласно настройкам Trigger.dev, но не должен стартовать параллельно. Это дороже, чем быстрый штатный запуск, поэтому реализация должна иметь сетевые timeout'ы на внешние запросы и завершаться быстро.

## 12. Стоимость и модель выполнения

Целевая модель выполнения: много коротких run'ов вместо одного долгого процесса.

Запрещено:

- запускать бесконечный цикл `while (true)` внутри Trigger.dev;
- держать процесс живым через `setInterval`;
- использовать `wait.for("60s")` для собственного цикла;
- делать Telegram long polling с ожиданием новых сообщений;
- self-trigger'ить задачу в конце run'а вместо cron-расписания.

Разрешено:

- один cron-trigger раз в минуту;
- один короткий `getUpdates` за запуск;
- один цикл по подписчикам внутри текущего запуска для отправки сообщений;
- быстрые HTTP-запросы к PADL API с явными timeout'ами;
- завершение run сразу после сохранения state.

Для контроля стоимости в логах каждого запуска нужно фиксировать:

- время старта и завершения run;
- количество подписчиков;
- количество найденных слотов;
- количество отправленных сообщений;
- длительность поиска слотов;
- длительность рассылки.

## 13. Границы ответственности модулей

Рекомендуемая структура будущей реализации:

- `config`: чтение и валидация env.
- `state-store`: чтение и запись Trigger.dev Session metadata.
- `telegram`: `getUpdates`, разбор `/start`, `sendMessage`, классификация ошибок отправки.
- `padl-api`: получение площадок, карточек игр, дат и availability.
- `slot-filter`: нормализация и фильтрация слотов.
- `message-format`: группировка слотов по площадкам и сборка Telegram-текста.
- `tasks/padl-monitor`: scheduled task, который связывает все модули.

Каждый модуль должен быть тестируем отдельно без запуска Trigger.dev.

## 14. Масштаб и лимиты хранения

Хранение подписчиков в Trigger.dev Session metadata рассчитано на небольшой список подписчиков: личный бот, команда, небольшой закрытый круг пользователей. Это соответствует ограничению "без сторонних сервисов".

Если бот должен обслуживать сотни или тысячи пользователей, metadata одной session может стать слишком маленькой или неудобной для конкурентного обновления. В таком масштабе обычно нужна внешняя база данных, но она сознательно не входит в текущую архитектуру из-за исходного ограничения проекта.

## 15. Тестирование

Минимальные тесты:

- env parsing: корректные значения, пустые значения, неверное время, неверное количество людей;
- state-store: добавление подписчика, повторный `/start`, удаление заблокированного подписчика, сохранение offset;
- Telegram update parsing: `/start`, `/start payload`, некомандный текст, отсутствующий `message`;
- slot-filter: строгое равенство `available_tickets === PADL_REQUIRED_PEOPLE`;
- slot-filter: границы времени `PADL_TIME_FROM` и `PADL_TIME_TO` включительно;
- message-format: пустой список, группировка по площадкам, разделение длинного сообщения;
- task orchestration: ошибка поиска слотов не удаляет подписчиков; ошибка одного Telegram send не останавливает рассылку.
- task lifecycle: задача не содержит бесконечных циклов, не вызывает `wait.for()` для минутного ожидания и завершает run после одной проверки.
- Telegram polling: `getUpdates` вызывается с `timeout: 0` в production-конфигурации.

Интеграционная проверка:

- локальный dry-run с моками Telegram и PADL API;
- проверка реального `getUpdates` на тестовом боте;
- проверка одного реального запуска Trigger.dev в dev/staging окружении.

## 16. Что не входит в первую версию

- Бронирование слотов.
- Команда `/stop`.
- Индивидуальные настройки пользователей.
- Админ-панель.
- Webhook-интеграция Telegram.
- Внешняя база данных.
- Массовая публичная рассылка на сотни или тысячи подписчиков.
- Постоянно работающий daemon-процесс.
- Бесконечный Telegram polling-loop.
